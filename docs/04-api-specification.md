# 04 — API Specification

REST, JSON, versioned under `/v1`. OpenAPI 3.1 generated by NestJS Swagger at `/v1/openapi.json` and served as docs at `/v1/docs`. All endpoints (except auth + Stripe webhook) require a verified identity and an active org membership.

## 1. Auth

- **User sessions:** Clerk (or Auth.js) issues a JWT. API verifies signature against the provider JWKS, extracts `externalId`, resolves `User` + active `Membership` → `{ userId, orgId, role }`.
- **Machine access:** `Authorization: Bearer glk_live_…` API keys. Stored as `sha256(key)`; only a prefix is shown after creation.
- **Headers:** `X-Org-Id` optional override when a user belongs to multiple orgs (validated against memberships).

## 2. Conventions

- **Pagination:** cursor-based. `?limit=50&cursor=<opaque>`. Response: `{ data: [...], nextCursor: string | null }`.
- **Filtering:** `?websiteStatus=NO_WEBSITE,BROKEN&temperature=HOT&minRating=4&city=Miami&hasEmail=true&minOpportunity=60`.
- **Sorting:** `?sort=newest|oldest|rating_desc|rating_asc|reviews_desc|opportunity_desc`.
- **Errors:** RFC 9457 problem+json: `{ type, title, status, detail, traceId }`.
- **Idempotency:** mutating POSTs accept `Idempotency-Key`; required on `/ingest`.
- **Rate limits:** see §6. Responses include `X-RateLimit-Remaining/Reset`.

## 3. Endpoint surface

### Campaigns
| Method | Path | Notes |
|--------|------|-------|
| GET | `/v1/campaigns` | list, search, sort |
| POST | `/v1/campaigns` | create (enforces plan campaign limit) |
| GET | `/v1/campaigns/:id` | with rollups |
| PATCH | `/v1/campaigns/:id` | rename / edit query / status |
| POST | `/v1/campaigns/:id/duplicate` | |
| POST | `/v1/campaigns/:id/archive` | sets `ARCHIVED` |
| DELETE | `/v1/campaigns/:id` | cascade (audited) |

### Businesses (leads)
| Method | Path | Notes |
|--------|------|-------|
| GET | `/v1/campaigns/:id/businesses` | the MAIN TABLE: filter + sort + cursor |
| GET | `/v1/businesses/:id` | full card incl. breakdown, contacts |
| PATCH | `/v1/businesses/:id` | stage, owner, tags, customFields |
| POST | `/v1/businesses/bulk` | bulk: tag / stage / assign / delete / enrich |
| POST | `/v1/businesses/:id/notes` · `/tasks` | CRM |
| GET | `/v1/businesses/:id/activity` | timeline |

### Ingest (extension)
| Method | Path | Notes |
|--------|------|-------|
| POST | `/v1/campaigns/:id/ingest` | batch of scraped businesses; dedups; updates `ScrapeJob`; **requires `Idempotency-Key`** |
| POST | `/v1/scrape-jobs` | create a job (returns id the extension reports progress to) |
| PATCH | `/v1/scrape-jobs/:id` | progress / pause / resume / cancel |

### Enrichment
| Method | Path | Notes |
|--------|------|-------|
| POST | `/v1/businesses/:id/enrich` | enqueue contact enrichment |
| POST | `/v1/campaigns/:id/enrich` | "Scrape Contacts" for selection/filter |
| GET | `/v1/businesses/:id/contacts` | results + confidence |

### Exports
| Method | Path | Notes |
|--------|------|-------|
| POST | `/v1/exports` | `{ campaignId, format, fields[], filters }` → enqueues render |
| GET | `/v1/exports/:id` | status + signed S3 URL when ready |

### Outreach
`/v1/sequences`, `/v1/sequences/:id/steps`, `/v1/sequences/:id/enroll`, `/v1/sequences/:id/stats`, plus tracking pixels `/t/open/:token.gif`, `/t/click/:token`, and `/u/:token` unsubscribe.

### Analytics
`GET /v1/analytics/overview` → the dashboard widgets (total campaigns, leads, websiteless count, emails/phones found, hot leads, conversion rate, export count, revenue). `GET /v1/analytics/timeseries?metric=…&range=…` for charts. `GET /v1/analytics/map` for the geo heat layer.

### Billing
`POST /v1/billing/checkout`, `POST /v1/billing/portal`, `GET /v1/billing/usage`, `POST /v1/billing/webhook` (Stripe; raw-body, signature-verified, no auth guard).

### Platform
`/v1/members` (invite/role), `/v1/api-keys`, `/v1/webhooks`, `/v1/audit-logs`, `/v1/tags`.

## 4. DTO & validation

DTOs are Zod schemas in `packages/types`, reused for (a) runtime validation via a NestJS Zod pipe and (b) the generated OpenAPI + typed client. Example:

```ts
export const IngestBusiness = z.object({
  placeId: z.string().optional(),
  googleCid: z.string().optional(),
  name: z.string().min(1),
  category: z.string().optional(),
  phone: z.string().optional(),
  website: z.string().url().optional(),
  addressLine: z.string().optional(),
  city: z.string().optional(), country: z.string().optional(),
  lat: z.number().optional(), lng: z.number().optional(),
  rating: z.number().min(0).max(5).optional(),
  reviewCount: z.number().int().nonnegative().optional(),
  hours: z.record(z.any()).optional(),
  photos: z.array(z.string().url()).max(20).optional(),
  attributes: z.record(z.any()).optional(),
});
export const IngestBatch = z.object({ businesses: z.array(IngestBusiness).max(200) });
```

## 5. RBAC

Guard order: `AuthGuard → TenantGuard → RolesGuard → RateLimitGuard`. `@Roles('ADMIN','MANAGER')` decorators + a central permission matrix (see [09](09-billing-rbac.md)). Object-level checks (owner-only edits) live in services.

## 6. Rate limiting

Redis sliding-window per `(orgId, route-class)`:

| Route class | Default limit |
|-------------|---------------|
| Read (GET) | 600 / min / org |
| Mutations | 120 / min / org |
| `/ingest` | 60 batches / min / org (≤200 rows each) |
| Enrichment enqueue | plan-scoped daily quota |
| Export render | 10 / hour / org |
| Auth-less (webhooks) | per-IP 100 / min |

External-provider calls (Hunter/Apollo/PSI/…) are rate-limited **inside the worker** with BullMQ limiters to respect vendor quotas, independent of API limits.
